# LongNovelInsight v0.1.0 — Release Baseline

> **Release Date:** 2026-05-18
> **Status:** Complete. All planned features implemented and verified.
> **Tag:** `v0.1.0`

---

## Goals

Build a local-first, single-user web application that uses LLMs to analyze a single long-form novel (.txt) and produce six types of structured literary insights — all data stored locally, no cloud dependencies.

### What was delivered

- Configure an LLM provider (DeepSeek or any OpenAI-compatible API) with built-in preset catalog
- Create a Topic, upload one .txt novel, auto-detect encoding (6 codecs), normalize to UTF-8
- Parse into chapters and overlapping chunks with whitespace normalization
- Run 6-type structured analysis via async parallel jobs with bounded concurrency
- Browse analysis outputs with evidence quotes, source chunk IDs, and confidence scores
- Ask free-form questions in a chat grounded in analysis outputs and source chunks
- View per-model token usage statistics from real LLM response data
- All data stored locally in SQLite + `data/` directory

## Architecture Summary

```
Browser (localhost:5173)
  → React 18 + TypeScript 5 + Vite 6
  → TanStack Query 5 (server state)
  → React Router 7.15
  → fetch → Backend (localhost:8000)
       → FastAPI + SQLModel + SQLite
       → OpenAI-compatible LLM (DeepSeek by default)
       → ThreadPoolExecutor (async parallel analysis)
```

- **11 database tables** across 2 SQLite tablespaces
- **38+ API endpoints** across 11 router modules
- **162 pytest tests** (all mock LLM, all passing)
- **3-panel chat UI** with draggable dividers and collapsible panels

## Analysis Pipeline

```
Parse → 6 async parallel LLM calls → Store outputs
         ↑                            ↑
    ThreadPoolExecutor          SQLite analysis_output
    (bounded 1–6, default 3)    + job metadata
```

- **6 output types**: overview, characters, relations, events, causality, themes
- **Per-type token budgets**: overview 1024, characters 3072, relations 2048, events 3072, causality 2048, themes 1536
- **Retry**: max 2 per type (429/5xx/timeout), JSON parse failures retry with doubled max_tokens
- **Prompt-cache-friendly**: shared system instructions + chunks form stable prefix, type-specific schema appended last
- **Batch-map-merge**: for novels exceeding context window, partial analysis per batch then multi-level merge
- **Effective config**: Topic override > Provider default > Preset default at runtime

## Known Limitations (v0.1.0)

| Limitation | Detail |
|-----------|--------|
| .txt only | No .epub or PDF support |
| Single novel per Topic | No cross-novel analysis |
| Chunk text in SQLite | May impact DB performance with very large novels; migration to disk files planned for v0.2 |
| Direct analysis repeats chunks | Selected chunks sent once per output type (6×); map-reduce pipeline planned for v0.2 |
| Keyword retrieval | Chat uses substring + stopword-based retrieval; no vector embeddings |
| No streaming | All LLM responses are request/response; no SSE or streaming |
| Single-user | No authentication, multi-user, or access control |
| Provider-specific thinking mode | Only DeepSeek `extra_body` supported; other providers may not honor it |
| Token usage estimates in chat | Chat tokens are real (from API response), but API may not always return all fields |
| No i18n | UI is English-only |
