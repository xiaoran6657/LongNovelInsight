# LongNovelInsight v0.2.0 — Scale & Depth for Single TXT

> **Release Date:** 2026-05-26
> **Status:** Complete. All planned features implemented and verified.
> **Tag:** `v0.2.0`

---

## Goals

Upgrade the analysis engine from v0.1's "6 types × N chunks" repeated-LLM-call model to a staged map-reduce pipeline: each chunk is sent to the LLM once for atom extraction, then deterministic merge and final output stages run in pure Python. Achieve ~4× token savings for large novels while adding run history, chunk range selection, analysis mode selection, provider config overrides, and E2E test coverage.

### What was delivered

- Staged analysis pipeline: local_extraction (LLM per chunk) → deterministic merge (Python) → final outputs (Python)
- Stable entity/event IDs generated backend-side with dedup across chunks
- Async job-first analysis with progress polling, cancel, retry-failed, and resume
- Preview / Range / Full / Incremental analysis modes with chunk/chapter range selection
- Run history with token usage tracking, per-stage progress bars, and failed-extraction details
- Provider config UX: per-topic overrides for model, max tokens, temperature, thinking mode, parallelism
- Max Tokens slider with min/value/max inputs and adaptive step size
- SQL-level pagination with `useInfiniteQuery` for large run lists
- 27 Playwright E2E tests with full API mocking covering the v2 analysis UI

## Architecture Summary

```
Analysis Pipeline (v2)
  Stage 1: local_extraction (per chunk, parallel)
    → local_characters, events, relations, causal_links, theme_signals,
      worldbuilding, foreshadowing_signals, open_questions

  Stage 2: type-specific merge (per type, deterministic Python)
    → dedup, alias resolution, canonical IDs, cross-chunk reference resolution

  Stage 3: final outputs (same 6 output types, from merged atoms)
    → overview, characters, relations, events, causality, themes

Storage
  Small JSON (≤64KB): inline SQLite analysis_output rows
  Large JSON (>64KB): disk files under data/topics/{id}/artifacts/
    tracked via analysis_artifact table with SHA256 hash
```

- **12 database tables** (6 new: AnalysisRun, LocalExtraction, ExtractedAtom, EntityRegistry, AnalysisArtifact, AnalysisOutputVersion)
- **44 API endpoints** across 11 router modules
- **406 pytest tests** (all LLM calls mocked), 1 integration test (deselected by default)
- **27 Playwright E2E tests** with full API mocking (5 smoke, 2 topic-detail, 20 v2 analysis UI)
- **47 frontend source files** (33 TSX components, 14 TS modules)

## Implemented Features

### Backend (Python + FastAPI)

| Area | v0.1.0 | v0.2.0 |
|------|--------|--------|
| Analysis Pipeline | 6 types × N chunks (6 LLM calls per chunk) | Staged map-reduce: 1 LLM call per chunk → merge → final |
| Token Cost | Scales with `novel_size × 6` | Scales with `novel_size` (~4× savings) |
| Entity IDs | LLM-generated, per-type | Stable backend-generated IDs with cross-type references |
| Job Model | Sync `POST /analysis/run` | Async `POST /analysis/runs` with 202 + polling |
| Analysis Modes | Fixed per-type | Preview/Range/Full/Incremental |
| Chunk Management | All-or-nothing | Chunk/chapter range selection, chunks/meta endpoint |
| Run History | None | Full history with token tracking, stage progress, retry/resume |
| Storage | All JSON inline in SQLite | Hybrid: ≤64KB inline, >64KB on disk with artifact tracking |
| Provider Config | Provider-level only | 3-layer resolution: Topic override > Provider default > Preset default |
| Output Versioning | Latest only | Run-scoped outputs with version history |
| Max Tokens Control | Static per-type budgets | Dynamic slider with per-topic override |
| ID System | None | Stable slugs: pinyin characters, title-based events |

### New Backend Services

| Service | Purpose |
|---------|---------|
| `analysis_run_service.py` | Run lifecycle orchestration, parallel extraction, merge, cancel |
| `local_extraction_worker.py` | Per-chunk LLM atom extraction with retry |
| `merge_service.py` | Deterministic type-specific merge with dedup |
| `final_output_service.py` | Final output generation from merged atoms |
| `atom_normalizer.py` | Stable ID generation and atom normalization |
| `artifact_service.py` | Hybrid storage management (inline ↔ disk) |

### Frontend (React + TypeScript + Vite)

| Feature | Description |

| Chunk Range Selector | By chunk index or chapter index, with validation |
| Run Panel | Create, monitor progress, cancel, view stage bars |

## Known Limitations (v0.2.0)

| Limitation | Detail |
|-----------|--------|
| .txt only | EPUB parsing planned for v0.3 |
| 6 output types | Same types as v0.1; worldbuilding and foreshadowing have merge support but no final output builders yet; timeline and character_arcs not yet implemented |
| Single novel per Topic | Multi-book Topics planned for v0.4 |
| Keyword retrieval | Chat retrieval still primarily keyword-based; FTS5 and hybrid retrieval planned for v0.3 |
| No streaming | All LLM responses are request/response |
| Single-user | No authentication or multi-user support |
| Provider-specific thinking | Only DeepSeek `extra_body` supported |
| Old API preserved | v1 `/analysis/run`, `/analysis/jobs`, `/analysis/outputs` endpoints kept for backward compatibility |

## Breaking Changes from v0.1.0

- **Database schema**: v0.2 adds 6 new tables. v0.1 databases are auto-migrated on startup. Old v0.1 tables are preserved — no data loss.
- **API additions**: v2 endpoints under `/topics/{id}/analysis/runs` and `/analysis/runs/{id}`. All v1 endpoints continue to work.
- **Frontend**: TopicDetailPage split into feature modules. Analysis UX redesigned around v2 pipeline. Legacy v1 analysis panel still available at the bottom of the page.

## Upgrade from v0.1.0

1. `git checkout v0.2.0`
2. Restart backend — `init_db()` auto-migrates the existing SQLite database
3. Restart frontend — `npm install && npm run dev`
4. Existing Topics, Providers, Documents, and analysis outputs are preserved
5. New v2 analysis runs coexist with legacy v1 jobs
