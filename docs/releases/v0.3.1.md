## Goals

Fix the top causes of v2 analysis extraction failures and misleading token reporting, based on real-world EPUB testing feedback. Make the pipeline more resilient to transport errors, output truncation, and thinking-mode token inflation; align cost estimates with actual usage; and provide accurate cumulative token accounting that reflects all LLM attempts, not just the last one.

## What Changed

### 1. LLM Transport Error Resilience

**Problem:** `httpx.RemoteProtocolError`, `peer closed connection`, and `incomplete chunked read` errors bubbled up as bare exceptions, causing entire runs to fail with opaque error messages.

**Fix:** `llm_client.py` now catches `httpx.TransportError` (the parent class of all transport-level exceptions) and wraps it as `LLMClientError`, preserving retry behavior and masking API keys. Transport errors are classified as retryable alongside existing timeout/network errors.

### 2. Truncation Detection & finish_reason Tracking

**Problem:** DeepSeek `max_tokens=8192` with thinking mode enabled frequently produced truncated JSON output. The pipeline had no way to distinguish truncation from malformed JSON, so retry strategies were suboptimal.

**Fix:**
- `LLMResponse` and `LocalExtractionResult` now carry `finish_reason` from the API response.
- When `finish_reason == "length"` or `completion_tokens >= max_tokens - 8`, the error message includes diagnostic text: `"likely truncated at max_tokens=8192"`.
- This lets the UI and retry logic distinguish budget exhaustion from format errors.

### 3. Adaptive Retry Token Escalation

**Problem:** The old retry strategy doubled `max_tokens` but capped at 8192 â€” which was already the user's configured value. Chunks stuck at 8192 would never succeed.

**Fix:**
- `RETRY_MAX_TOKENS` raised to 16384.
- Token escalation per attempt: `original â†’ min(original Ă— 2, 16384) â†’ 16384`.
- Default `max_tokens` raised from 3072 to 4096 to reduce initial truncation risk.
- Error classification re-evaluated per retry attempt (not once before the loop), so a transport error on attempt 1 followed by JSON truncation on attempt 2 correctly escalates tokens on attempt 3.
- 429 rate-limit errors use longer backoff (15sâ€“30s); JSON truncation uses short backoff (1.5sâ€“3s); transport errors use moderate backoff (3sâ€“6s).
- Additional retryable keywords: `timeout`, `timed out`, `transport error`, `network error`, `peer closed`, `incomplete chunked read`.

### 4. Thinking Mode Awareness

**Problem:** Users with `thinking_mode=enabled` in their topic override experienced high failure rates because reasoning tokens consumed the `max_tokens` budget, leaving little room for JSON output. The pipeline gave no indication this was happening.

**Fix:**
- `provider_default` thinking mode sends no `thinking` extra_body (unchanged â€” provider decides).
- `thinking_mode=enabled` adds a warning to the extraction result: *"thinking mode enabled for local extraction may increase latency and output size risk"*.
- Cost estimates apply a 1.4Ă— output buffer when thinking mode is enabled.
- Users are not forced to disable thinking; the system surfaces the risk without overriding their choice.

### 5. Prompt Output Limits

**Problem:** The `local_extraction.md` prompt had no per-category atom limits, so LLMs could produce very large JSON arrays that exceeded `max_tokens`, causing truncation.

**Fix:** Added explicit per-chunk limits to the prompt:

| Atom type | Max items |
|-----------|-----------|
| local_characters | 8 |
| local_events | 8 |
| local_relations | 8 |
| local_causal_links | 6 |
| local_theme_signals | 6 |
| local_worldbuilding | 6 |
| local_foreshadowing | 2 |
| local_open_questions | 4 |

Priority ordering: characters > events > relations > causality > themes > worldbuilding > open_questions. Evidence quotes limited to 1â€“2 per atom, under 80 characters each.

### 6. Causality Resolved Logic

**Problem:** `merge_causality` compared free-text `cause_hint`/`effect_hint` directly against event `stable_id` strings, producing near-100% unresolved rates even on full runs.

**Fix:** New multi-strategy matching:
- Builds an event index from `stable_id`, `title`, `summary`, and `id_hint` fields from `content_json`.
- Matches via direct equality, normalized equality, substring containment (minimum 4 characters normalized to avoid false positives).
- Resolved links emit `resolved_cause_event_id` / `resolved_effect_event_id` in the merged output.
- Short strings (< 4 chars) are never matched to avoid false positives.

### 7. Final Output Warning Consolidation

**Problem:** `build_final_causality` created one warning per unresolved causal link. With 50+ unresolved links, the warnings array exploded.

**Fix:** Consolidated into a single summary warning: *"N unresolved causal links in final output; examples: ..."* (capped at 5 examples). Each item's `resolved` boolean is preserved for the frontend.

### 8. Run Status Protection

**Problem:** A late background exception after the pipeline completed (e.g., `_fail_run` called after merge/final stages succeeded) could overwrite a completed run's status from `succeeded`/`partial_success` to `failed`.

**Fix:** `_fail_run` now refuses to overwrite runs with status `SUCCEEDED`, `CANCELLED`, or `PARTIAL_SUCCESS`, and runs whose `finished_at` is set and `metadata.stage == "completed"`.

### 9. Cumulative Token Accounting

**Problem (major):** The pipeline only tracked the *last* LLM attempt's token usage. All failed retry attempts' tokens were lost from the run statistics, causing a 40â€“60% gap between recorded tokens and the provider's billing dashboard.

**Fix:**
- New `AttemptUsage` dataclass captures per-attempt token usage including `reasoning_tokens`, `prompt_cache_hit_tokens`, and `prompt_cache_miss_tokens`.
- `LocalExtractionResult` accumulates `cumulative_prompt_tokens`, `cumulative_completion_tokens`, `cumulative_total_tokens`, `cumulative_reasoning_tokens`, and cache fields across all attempts.
- `LocalExtraction` model gains persistent columns: `reasoning_tokens`, `prompt_cache_hit_tokens`, `prompt_cache_miss_tokens`, `usage_unavailable_attempts`, `attempt_usage_json`.
- `AnalysisRun` now tracks `prompt_tokens` and `completion_tokens` in addition to `total_tokens`.
- `get_analysis_run_status` returns the full breakdown.
- Attempts where the API returns no usage (transport errors, 429 without response body) are marked `usage_available=false`; `usage_unavailable_attempts` is incremented without fabricating token counts.

**DeepSeek cache fields:** The worker reads `prompt_cache_hit_tokens` / `prompt_cache_miss_tokens` directly from the DeepSeek usage response, with fallback to OpenAI-style `prompt_tokens_details.cached_tokens`.

**Retry/resume persistence:** `retry_failed_extractions` and `resume_analysis_run` now save cumulative usage fields and recompute run-level totals via `_recalculate_run_usage_from_extractions()`. Token values are added to existing rows rather than replaced.

### 10. Cost Estimate Alignment

**Problem:** The frontend and backend cost estimates hardcoded `output = n_chunks Ă— 2048`, ignoring the user's `max_output_tokens` configuration, thinking mode, and retry risk. Predicted output was often 4Ă— lower than actual.

**Fix:**
- Backend `estimate_v2_analysis_cost` now accepts `max_output_tokens`, `thinking_mode`, `mode`, and `retry_multiplier`.
- Output formula: `n Ă— (min(max_output_tokens, 16384) Ă— 0.65) Ă— retry_multiplier`.
- Thinking mode enabled â†’ additional 1.4Ă— buffer.
- Retry multiplier: preview/incremental 1.15, full/range 1.25.
- Merge and final stages correctly reported as zero LLM cost (deterministic Python).
- Frontend `analysisSelection.ts` uses the same formula, reading `EffectiveProviderConfig` from the active topic.
- `AnalysisCostProjection` disclaimer updated: *"Estimated API usage, includes retry risk buffer. Provider dashboard may be higher if failed/timeout attempts have server-side usage not returned by API."*

### 11. UI Token Breakdown

**Problem:** The run detail panel showed a single "Tokens: X" line with no breakdown.

**Fix:** `AnalysisRunPanel` now displays:
```
Tokens: total X Â· input Y Â· output Z Â· reasoning R
```
When `usage_unavailable_attempts > 0`, an orange warning appears:
*"N failed attempt(s) did not return usage; provider dashboard may be higher."*

## Known Limitations (v0.3.1)

| Limitation | Detail |
|-----------|--------|
| Retry attempt_usage_json not merged | When retrying a failed extraction, the new `attempt_usage_json` replaces the old one rather than appending. Token totals are correct (cumulative via `+=`); full per-attempt audit history can be added in a future patch. |
| Thinking mode warnings are worker-local | The thinking-mode warning appears in `LocalExtractionResult.warnings` but is not yet persisted to the `AnalysisRun` metadata or UI. |
| Provider dashboard may still exceed local stats | API calls that fail mid-response (transport errors, timeouts) may have server-side usage that the API does not return. These are tracked as `usage_unavailable_attempts` with zero tokens. |
